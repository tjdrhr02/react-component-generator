# TDD Rules

> **이 규칙은 Rigid — 상황에 맞게 변형하지 마라.**
> 프로젝트의 AGENTS.md에 TDD 규칙이 별도로 존재하면 그것을 우선한다. 이 파일은 기본값(fallback)이다.

---

## 적용 기준

**TDD 필수 적용 대상:**
- 비즈니스 로직 (데이터 변환, 계산, 상태 전이)
- API 엔드포인트 핸들러
- 유틸리티 함수 (`stripCodeFences`, `ensureRenderCall` 등)
- 버그 수정 — 버그를 재현하는 테스트를 먼저 작성한 뒤 수정

**TDD 불필요 대상:**
- 타입 정의 (`types/index.ts` 등)
- 설정 파일 (`vite.config.ts`, `tsconfig.json`, `eslint.config.js`)
- 순수 UI 렌더링 (스타일, 레이아웃만 다루는 컴포넌트)
- SQL 마이그레이션 및 스키마 파일

## RED-GREEN-REFACTOR 사이클

### RED — 실패하는 테스트 작성

- **하나의 동작 = 하나의 테스트.** 여러 동작을 한 테스트에 묶지 마라.
- 테스트를 작성한 뒤 **반드시 실행하여 실패를 눈으로 확인**한다.
- 실패 이유는 **"기능이 아직 구현되지 않아서"** 여야 한다. 문법 오류, import 오류, 타입 오류로 실패하면 그것부터 고쳐라.
- 테스트 이름은 동작을 서술한다: `"유효하지 않은 provider가 들어오면 null을 반환한다"`.

### GREEN — 최소한의 코드로 테스트 통과

- **테스트를 통과시키는 가장 단순한 코드**만 작성한다.
- YAGNI — 지금 실패 중인 테스트가 요구하지 않는 기능을 미리 추가하지 마라.
- 신규 테스트뿐 아니라 **기존 테스트 전체가 통과하는지 확인**한다.
- 코드가 지저분해도 된다. 정리는 REFACTOR 단계에서 한다.

### REFACTOR — 동작 유지하며 구조 개선

- 중복 제거, 이름 개선, 헬퍼 함수 추출만 허용한다.
- **새로운 동작을 이 단계에서 추가하지 마라.** 새 동작이 필요하면 다음 RED로 돌아가라.
- 리팩터링 전후로 **테스트 전체가 green 상태를 유지**하는지 확인한다.

### 반복

REFACTOR가 끝나면 다음 동작에 대한 RED로 돌아간다. 사이클을 건너뛰지 마라.

---

## 삭제 강제 규칙

테스트 전에 프로덕션 코드를 먼저 작성했다면:

1. **작성한 프로덕션 코드를 삭제한다.**
2. RED 단계부터 재시작한다.

"참고용으로 주석 처리해두는 것"도 금지한다. 삭제하지 않으면 RED 단계를 건너뛰게 되어 사이클이 오염된다.

## 변명 차단표

| 변명 | 반론 |
|---|---|
| "너무 단순해서 테스트가 불필요하다" | **단순한 코드가 가장 먼저 망가진다.** 단순함은 테스트를 생략할 이유가 아니라 테스트 작성 비용이 낮다는 뜻이다. |
| "나중에 테스트를 추가하겠다" | **나중은 오지 않는다.** 코드가 완성된 후 테스트를 추가하는 것은 TDD가 아니며, 설계 피드백도 얻을 수 없다. |
| "시간이 없다" | **테스트 없이 작성한 코드는 디버깅 시간을 더 많이 소모한다.** TDD는 속도를 줄이는 것이 아니라 재작업을 줄인다. |
| "코드를 삭제하면 낭비다" | **잘못된 순서로 작성된 코드는 이미 낭비다.** RED 없이 작성된 코드는 테스트로 검증된 것이 아니다. |
| "프로토타입이라 괜찮다" | **프로토타입은 버려진다. 버려지지 않으면 프로토타입이 아니다.** 프로덕션으로 살아남은 순간 이 규칙이 적용된다. |
